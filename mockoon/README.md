# ACITS — Mockoon mock server

Моковый бэкенд для приложения **acits_flutter**. Повторяет формы ответов боевого
`app.acits.ru` (Django REST + SimpleJWT) с красивым связным датасетом и **двумя
организациями**, между которыми можно переключаться прямо в приложении.

| Приют | id | Город | Животные |
|---|---|---|---|
| «Добрые лапы» | 1 | Москва | 8 (Рекс, Мурка, Барон, Ласка, Джек, Соня, Тайсон, Симба) |
| «Верный друг» | 2 | Санкт-Петербург | 6 (Граф, Багира, Локи, Ася, Тор, Феня) |

У каждого приюта свои кураторы, заявители, усыновители, назначения, заметки,
аптека и статистика. Пользователь **Мария Администратова** — админ обоих.

## Запуск

**Через приложение Mockoon (GUI):**
1. Открыть Mockoon → `File → Open environment` → выбрать `mockoon/mockoon.json`.
2. Нажать ▶️ Start. Сервер поднимется на `http://localhost:3000`.

**Через CLI:**
```bash
npx @mockoon/cli start --data mockoon/mockoon.json --port 3000
```

## Как направить приложение на мок

Базовый адрес контуров лежит в `lib/domain/env.dart` (`AcitsEnvUrls`).
Для локального мока укажи `http://10.0.2.2:3000` (Android-эмулятор) или
`http://localhost:3000` (iOS-симулятор/десктоп) вместо боевого хоста —
например, добавив пункт в список окружений debug-экрана или временно в
`EnvRegistrer.createEnv()`.

## Переключение между организациями

Приложение шлёт заголовок `x-current-shelter: <id>` на все запросы данных.
Мок отдаёт **разные данные** в зависимости от него (правило по заголовку в
каждом маршруте):

- `x-current-shelter: 1` → «Добрые лапы» (Москва);
- `x-current-shelter: 2` → «Верный друг» (СПб);
- без заголовка → приют 1 (default-fallback, чтобы ничего не падало).

В самом приложении переключение штатное: экран выбора приюта дёргает
`GET /api/v1/users/me/shelters/current/` с нужным `x-current-shelter`, после чего
весь UI (животные, назначения, кураторы, статистика) показывает данные выбранной
организации. Список организаций пользователя отдаёт `GET /api/v1/users/me/shelters/`.

> Быстрая проверка из терминала:
> ```bash
> curl -s -H "x-current-shelter: 2" localhost:3000/api/v1/animals/ | jq '.results[].name'
> ```

## Что внутри (79 маршрутов)

| Группа | Эндпоинты |
|---|---|
| Auth | `POST /api/token/`, `POST /api/token/refresh/` |
| Профиль | `GET/PATCH /api/v1/users/me/`, `.../change_password/` |
| Сброс пароля / верификация | `users/reset-password/`, `.../complete/`, `.../confirm/:uidb64/:token/`, `users/verify-email/…`, `users/verify-worker/…` |
| Приюты | `me/shelters/`, `me/shelters/current/`, `available-shelters/`, `shelters/`, `shelters/:id/` |
| Животные | `animals/`, `animals/:id/`, `species/`, `attributes/`, `stats/`, `:id/files/`, `:id/history/`, `:id/restore/` |
| Усыновления / Передержки / Выпуски | `animals/:id/adoptions/`, `.../overstays/`, `.../releases/` (+ `:id/`) |
| Заметки | `animals/notes/`, `animals/notes/:id/` |
| Кураторы / Заявители / Усыновители / Передержчики | `curators/`, `applicants/`, `adopters/`, `animal_sitters/` (+ `:id/`) |
| Назначения | `prescriptions/`, `prescriptions/:id/`, `prescriptions/executions/` |
| Приют | `shelter/drugs/`, `shelter/workers/` (+ approve/decline) |
| Справочники | `values-for-selection/`, `feedback/` |

Покрытие сверено с живой схемой `dev-01.app.acits.ru/api/schema/` — все 58 путей контракта
представлены (вложенные под-ресурсы животного, передержчики и flow сброса пароля/верификации
добавлены к базовому набору).

- **GET-списки** — пагинация DRF: `{count, next, previous, results[]}`.
- **Детали по `:id`** — правило `x-current-shelter` AND `id` возвращает нужный
  объект нужной организации, с fallback на первый объект приюта 1.
- **Shelter-зависимые** маршруты (животные, кураторы, заявители, усыновители,
  назначения, заметки, аптека, работники, статистика, `shelters/:id`,
  `me/shelters/current`) — по варианту ответа на каждый приют.
- **Глобальные** (одинаковые для всех): `species`, `attributes`,
  `values-for-selection`, `me`, `me/shelters`, `available-shelters`, auth.
- **Мутации** (POST/PATCH/PUT/DELETE) — заглушки: `201`/`200`/`204` с эхом объекта.

## Проверка ошибок

У **каждого** маршрута кроме happy-path добавлены error-варианты ответа с точными
формами боевого бэкенда (сняты с dev/prod). По умолчанию активен успешный ответ;
чтобы проверить обработку ошибки — открой маршрут в Mockoon и **кликни по нужному
error-варианту** (он станет активным, зелёная точка). Ответ отдаётся сразу, без
перезапуска.

Варианты помечены по статусу и причине, например: `401 · неверные креды`,
`401 · битый токен`, `403 · нет x-current-shelter`, `404 · не найдено`,
`400 · валидация`, `429 · throttled`, `500 · server error`. Для гостевых ссылок из
письма (verify-email/verify-worker/reset-password) — `400 · invalid_link`,
`400 · bad_user`, `400 · bad_shelter` (статусы из `frontend_info_status`).

Точные тела (как у Django REST + SimpleJWT):

| Статус | Когда | Тело |
|---|---|---|
| 401 | неверный логин/пароль на `/api/token/` | `{"detail":"No active account found with the given credentials"}` |
| 401 | нет токена | `{"detail":"Authentication credentials were not provided."}` |
| 401 | битый/просроченный токен | `{"detail":"Given token not valid...","code":"token_not_valid","messages":[...]}` |
| 400 | не заполнены поля логина | `{"username":["This field is required."],"password":[...]}` |
| 400 | невалидный POST животного | `{"spec_id":["This field is required."], ...}` |
| 403 | нет заголовка `x-current-shelter` | `{"detail":"Missing or bad current-shelter."}` |
| 403 | доступ к чужому приюту | `{"detail":"User ... does not exist in shelter with id N."}` |
| 404 | ресурс не найден | `{"detail":"Not found."}` |
| 429 | троттлинг | `{"detail":"Request was throttled..."}` |
| 500 | ошибка сервера | `{"detail":"Внутренняя ошибка сервера."}` |

Набор ошибок зависит от типа маршрута: auth-логин → 401/400/429/500; detail-GET →
404/401/403/500; создание → 400/401/403/500; verify/reset-flow → 400 invalid_link/bad_user/
bad_shelter + 500; и т.д. Всего ~317 error-вариантов на 79 маршрутах.

> После проверки не забудь вернуть активным успешный ответ (обычно первый в списке).

## Регенерация

Конфиг собран из живых ответов прода и лежащих рядом скриптов:

```bash
cd mockoon && python3 build.py
```

- `gen_mockoon.py` — датасет и тела ответов (правь данные тут).
- `build_env.py` — сборка Mockoon environment (маршруты, правила, CORS).
- `build.py` — запускает оба по очереди.

Скрипты самодостаточны (справочники `values-for-selection` вшиты). Можно также
редактировать тела прямо в Mockoon GUI — тогда правки вноси и в скрипты, иначе
следующая регенерация их перезатрёт.

> `_bodies.json` — промежуточный артефакт сборки, в VCS не нужен.
